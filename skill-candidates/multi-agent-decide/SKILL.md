# multi-agent-decide

사용자가 **결정**을 내려야 하는 모든 순간에, 한 명의 AI 머릿속에서 짠 옵션 3개가 아니라 **여러 에이전트가 각기 다른 관점에서 독립적으로 도출한 옵션 + 그 옵션들에 대한 토론 결과**를 제시한다. 사용자가 받는 것은 "AI가 골라준 단일 답"이 아니라 "여러 관점이 충돌한 뒤 합의된 추천안 + 소수의견".

## 언제 발동되는가

- **자동 (기본)**: AI가 사용자에게 옵션 중 하나를 고르라고 물어야 하는 모든 상황. 예:
  - 아키텍처/설계 선택 (DB 스키마, API 형태, 컴포넌트 구조 등)
  - 도구·라이브러리·프레임워크 선택
  - 다음 작업 슬라이스 결정 ("Slice 3 후보 중 무엇을 할까")
  - 리팩토링 범위·전략 선택
  - 마이그레이션·롤아웃 전략
- **수동**: `/multi-agent-decide <topic>` 으로 명시 호출.
- **발동 안 함**:
  - 단순 확인/yes-no ("이대로 커밋할까요?")
  - 명백한 사실 질문 (값을 묻는 정도)
  - 이미 답이 정해진 사소한 분기 (파일명, 변수명 1-2글자 차이 등)
  - 사용자가 이미 결정을 내려준 상황

## 동작 절차

### 1. 결정 프레이밍 (메인 에이전트)

사용자에게 옵션을 묻기 **전에**, 결정 문제를 한 문단으로 정리한다:

- **무엇을 결정하는가** (한 문장)
- **이 결정에 따라 무엇이 달라지는가** (영향 범위)
- **현재까지의 제약 조건** (이미 정해진 것, 트레이드오프, 시간/비용 한계)

이 프레이밍이 그대로 각 에이전트에게 전달되는 컨텍스트가 된다.

### 2. 1라운드 — 병렬 옵션 제안 (최소 3명 + 가능한 한 더 많이)

`Agent` 도구를 **한 메시지에 여러 tool_use 블록으로** 묶어 병렬 실행한다. 각 에이전트는 **서로 다른 관점**을 부여받는다. 결정의 성격에 맞게 관점을 고르되, 최소 3명, 가능하면 4-6명까지 늘린다.

관점 카탈로그 (결정 유형에 따라 골라 쓰기):

- **아키텍트** — 장기 유지보수성, 결합도, 추상화 계층
- **실용주의자 / 미니멀리스트** — 가장 짧은 경로, YAGNI, 지금 당장 필요한 만큼만
- **운영자 / SRE** — 배포·관측·롤백·온콜 부담
- **보안 담당** — 권한, 입력 검증, 공격 표면, 라이선스/컴플라이언스
- **성능 엔지니어** — 지연시간, 처리량, 비용 곡선
- **사용자 경험 옹호자** — 최종 사용자가 실제로 겪는 흐름
- **신입 온보딩 관점** — 다음 사람이 6개월 뒤에 읽었을 때 이해 가능한가
- **회의주의자 / Red Team** — 이 방향이 망하는 시나리오, 숨은 가정
- **비용 회계사** — 토큰/시간/인력/인프라 비용
- **테스트 가능성 옹호자** — 회귀 자산, 검증 비용

각 에이전트에게 보내는 프롬프트는 다음 구조:

```
당신은 <관점> 입장에서 다음 결정을 검토합니다.

[결정 프레이밍 — 1단계에서 정리한 것 그대로]

요청:
1. 이 관점에서 가장 합리적이라 보는 옵션 1개를 제안하라.
2. 왜 이 옵션이 다른 관점이 놓치기 쉬운 가치를 보전하는지 1-2문장.
3. 이 옵션의 가장 큰 약점/리스크 1가지를 정직하게 적어라.
4. 200자 이내 한국어로 보고.
```

각 에이전트는 독립적으로 실행되어 서로의 답을 보지 않는다 — 이게 다양성의 핵심.

### 3. 2라운드 — 심화 토론 (재반박)

1라운드 결과를 모은 뒤, **같은 에이전트들을 다시 한 번 병렬 호출**한다. 이번 프롬프트에는 다른 에이전트들의 1라운드 제안이 포함된다:

```
당신은 <관점> 입장입니다. 다른 관점의 에이전트들이 다음 옵션을 제시했습니다:

- [관점 A]: <옵션 A 요약 + 약점>
- [관점 B]: <옵션 B 요약 + 약점>
- [관점 C]: <옵션 C 요약 + 약점>
...

요청:
1. 다른 옵션 중 당신 관점에서 가장 위험하다고 보는 것 1개와 그 이유.
2. 자기 옵션을 수정·보완할 점이 있다면 수정안 (없으면 "유지").
3. 모든 옵션을 본 뒤, 최종적으로 당신이 사용자에게 추천할 옵션 (자기 안일 필요 없음).
4. 200자 이내 한국어.
```

### 4. 조정자 합성 (메인 에이전트)

메인 에이전트가 다음을 합성한다:

- **합의 추천안 (1개)** — 2라운드에서 가장 많은 에이전트가 최종 추천한 옵션. 동수면 약점이 가장 적은 쪽.
- **반대 의견 (1-2개)** — 합의안과 다르게 추천한 에이전트의 입장. 이 사람들이 본 위험을 1-2문장으로 인용.
- **결정 트레이드오프** — 합의안을 고를 때 사용자가 받아들이는 비용 1-2가지.

### 5. 사용자에게 제시

`AskUserQuestion`(또는 [[ask-step-by-step]] 스킬이 있다면 그 형식)으로 옵션을 노출한다:

- 옵션 라벨에 합의 추천안이 가장 먼저 오고 "(Recommended)" 표기.
- description에는 그 옵션의 핵심 가치 + 어떤 관점이 밀었는지 짧게.
- 최소 3개 옵션. 합의안 외에도 반대 의견이 추천한 안을 살려서 노출.

답변과 함께 사용자에게 **소수 의견의 우려도 한 줄로** 같이 보여준다. "고른 안이 망하는 경우는 X일 때"가 머릿속에 남도록.

## 비용·시간 가이드

- 1라운드 + 2라운드 = 에이전트당 2회 호출. 3명이면 6회, 5명이면 10회.
- 각 에이전트 응답을 200자로 제한해 컨텍스트 폭발을 막는다.
- 결정의 무게에 따라 인원수 조절:
  - **가벼움** (네이밍, 작은 분기): 이 스킬 발동 X.
  - **중간** (라이브러리 선택, 슬라이스 선택): 3명 / 2라운드.
  - **무거움** (아키텍처, 마이그레이션 전략): 4-6명 / 2라운드.

## 회피해야 할 함정

1. **에이전트들이 같은 답을 내는 것** — 관점이 충분히 다르지 않거나 프롬프트가 비슷할 때 발생. 관점 카탈로그에서 **명백히 충돌하는 쌍**(예: 미니멀리스트 vs 아키텍트, 실용주의자 vs 회의주의자)을 일부러 섞을 것.
2. **메인 에이전트의 사전 결론을 컨텍스트에 흘리는 것** — 프레이밍은 사실/제약만 담고, "내 생각엔 X가 좋아 보인다" 같은 문장을 절대 넣지 말 것.
3. **2라운드를 1라운드 요약으로 변질시키는 것** — 2라운드는 반드시 "다른 옵션 중 가장 위험한 것"을 지목하게 해 충돌을 강제.
4. **사용자에게 토론 전체 로그를 덤프하는 것** — 사용자가 보는 것은 최종 옵션 + 한 줄 소수의견. 토론 과정은 메인 에이전트의 사고 흐름에만 남는다.
5. **자동 트리거 남발** — yes/no, 단순 확인, 값 입력 같은 데에는 발동하지 말 것. 옵션이 정말 3개 이상 의미 있게 존재할 때만.

## 출력 형식 (사용자에게 보이는 최종)

```
**결정**: <한 문장 요약>

**추천**: <합의안 라벨>
> <왜 — 가장 많은 관점이 동의한 이유 한 줄>

**다른 후보**:
- <옵션 B> — <어떤 관점이 밀었는지 + 핵심 장점 한 줄>
- <옵션 C> — <어떤 관점이 밀었는지 + 핵심 장점 한 줄>

**유의할 점 (소수의견)**:
> "<반대한 에이전트의 우려를 한 문장 인용>"

[AskUserQuestion 으로 선택지 제시]
```

## 다른 스킬과의 관계

- [[ask-step-by-step]] — 사용자에게 묻는 **형식**의 규칙. 이 스킬은 그 형식으로 묻기 **전에** 옵션을 어떻게 도출하느냐를 다룬다. 두 스킬은 충돌 없이 결합된다: multi-agent-decide로 옵션을 만들고, ask-step-by-step 형식으로 묻는다.
