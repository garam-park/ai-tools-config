# 작업 분석

## 개요

Notion, 로컬 작업 파일, 브랜치, 코드, PR을 변경하지 않고 작업 내용을 분석한다. 작업이 무엇을 요구하는지, 어떤 일이 필요해 보이는지, 무엇이 불명확한지, 다음 워크플로우가 무엇인지 근거 중심으로 간결하게 설명한다.

## 작업 출처

- 워크스페이스: `(주)이노팸`
- 데이터베이스: `솔루션 개발팀 작업`
- 데이터베이스 ID: `71842431-f19c-4f43-9df7-461805cf3895`
- 데이터 원본 ID, MCP가 지원하는 경우: `42e60fb3-5260-429b-8af4-ed28535f295b`
- 작업 ID 속성: `작업 ID` (`unique_id`, 접두사 `TSK`)
- 상태 속성: `상태`
- 종료 상태: `완료`, `PR완료(DEV)`, `보관됨`

## 작업 절차

1. 작업을 식별한다.
   - `TSK-3477`, `tsk-3477`, `3477`, Notion 작업 URL, 로컬 작업 파일 경로, 현재 브랜치에서 추론한 작업 ID를 받을 수 있다.
   - 후보 작업이 여러 개면 조회나 분석 전에 짧은 확인 질문을 하나만 한다.
2. ID나 URL이 있으면 Notion 작업을 가져온다. 작업을 수정하지 않도록 `--start` 없이 기존 헬퍼를 우선 사용한다.

```bash
python3 /Users/garam/.codex/skills/start-task/scripts/notion_task.py TSK-3477 --config .codex/config.toml
```

3. 분석 근거가 충분하도록 Notion 속성과 본문 블록을 읽는다. 작업 문맥에는 `작업 이름`, `작업 ID`, `상태`, `우선순위`, `태그`, `담당자`, 마감일, `요약`, `Description`, 링크, 체크리스트, 의미 있는 본문을 포함한다.
4. `tasks/backlog`, `tasks/in-progress`, `tasks/review-ready`에서 표시용 작업 ID, Notion 페이지 ID, 정확한 제목을 검색한다. 일치하는 파일은 보조 문맥으로만 읽고, 이동/수정/생성하지 않는다.
5. 작업 영향 범위를 이해하는 데 필요할 때만 저장소 문맥을 살핀다.
   - 구현 워크스페이스에서는 프로젝트 탐색을 위해 `README.md`를 읽는다.
   - 작업이 구체적인 모듈, 엔드포인트, 화면, 워크플로우를 언급할 때만 관련 `AGENTS.md`, `.claude/rules/`, API 문서, 주변 코드를 읽는다.
   - 작업 의미를 설명하는 데 필요하지 않으면 구현 수준의 탐색은 피한다.
6. 확인된 사실과 추론을 분리한다. 모호한 작업 문구를 해석할 때는 가정을 명시한다.
7. 다음 스킬이나 액션을 추천한다.
   - 요구사항이 불명확하거나 인수 기준을 작성해야 하면 `$spec-task`를 추천한다.
   - 작업이 실행 가능하고 현재 저장소에서 구현해야 하면 `$start-task`를 추천한다.
   - 별도 worktree에서 구현해야 하면 `$start-task-worktree`를 추천한다.
   - 구현이 이미 커밋됐거나 PR 생성 준비가 됐으면 `$create-pr`를 추천한다.
   - 기존 PR의 리뷰, CI, 후속 수정이 필요하면 `$handle-pr`를 추천한다.

## 응답 형식

사용자가 자세한 설명을 요청하지 않으면 짧게 답한다. 사용자의 언어를 따른다.

필요하면 아래 섹션을 포함한다.

- `작업 요약`: 작업 내용을 쉬운 말로 한두 문장으로 설명한다.
- `확인된 사실`: 상태, 우선순위, 담당자, 마감일, 출처 링크, 직접 명시된 요구사항을 정리한다.
- `해야 할 일`: 예상되는 변경 또는 조사 범위를 구체적으로 적는다.
- `영향 범위`: 관련될 가능성이 있는 저장소, 모듈, API, 화면, 데이터, 문서, 테스트를 적는다.
- `불명확한 점`: 막히는 모호함과 막히지는 않는 가정을 구분한다.
- `리스크`: 호환성, 마이그레이션, 권한, 회귀, 배포, 테스트 리스크를 적는다.
- `추천 다음 단계`: 보통 `$spec-task`, `$start-task`, `$start-task-worktree`, `$create-pr`, `$handle-pr` 중 하나를 직접 추천한다.

## 가드레일

- Notion 상태를 `진행 중`으로 바꾸지 않는다.
- Notion 페이지, 로컬 작업 파일, 제품 코드, 브랜치, 커밋, PR, GitHub 코멘트를 수정하지 않는다.
- 누락된 요구사항을 지어내지 않는다. 확실하지 않은 해석은 가정으로 표시한다.
- Notion 조회가 실패하면 사용 가능한 로컬 작업 문맥만 분석하고, 어떤 출처를 가져오지 못했는지 명확히 말한다.
- 작업이 터미널 상태이면 그 상태를 보고하고, 사용자가 명시적으로 재개나 후속 처리를 원하지 않는 한 구현을 추천하지 않는다.
