# ask-step-by-step

이 스킬은 AI가 사용자에게 질문할 때 따르는 **질문 방식의 기본 규칙**이다. 길고 장황한 복합 질문 대신, **작은 단위로 쪼갠 질문**을 **What / Why 라벨**과 함께 한 번에 하나씩 묻는다.

## 언제 발동되는가

- **자동 (기본)**: AI가 작업을 진행하다가 사용자에게 무언가를 묻는 모든 순간. 요구사항이 모호할 때, 설계 갈림길에 섰을 때, 위험한 작업 전 확인이 필요할 때, 옵션 중 하나를 골라야 할 때 등.
- **수동**: 사용자가 `/ask-step-by-step` 으로 명시 호출했을 때. 이 경우는 "지금부터 질문 방식을 이 모드로 유지하라"는 신호로 해석한다.

## 핵심 규칙

### 1. 한 번에 하나의 질문 단위

- 한 질문은 **하나의 결정**만 다룬다. 결정이 N개면 질문 턴도 N개로 나눈다.
- 한 메시지에 여러 질문을 묶지 않는다. 사용자의 답이 다음 질문의 전제가 되는 경우가 많으므로, 답을 받은 뒤에 다음 질문을 작성한다.
- 예외: 서로 **완전히 독립적**이고 사용자가 한꺼번에 처리하는 게 명백히 편한 경우에 한해 `AskUserQuestion` 도구로 최대 2~3개까지 묶을 수 있다. 의심스러우면 나누는 쪽을 택한다.

### 2. 질문은 헤딩, 이유는 인용 한 줄

각 질문은 아래 형식을 따른다:

```
### <결정해야 할 것 — 질문 형태로>
> <왜 이 결정이 필요한지, 답에 따라 무엇이 달라지는지 한 줄>

- <선택지 1 — 어떤 결과로 이어지는지>
- <선택지 2 — 어떤 결과로 이어지는지>
```

- **헤딩(`###`)** 에는 결정 대상을 질문 형태로 명확히 적는다. "어떻게 할까요?" 같은 모호한 표현은 금지.
- **인용(`>`)** 한 줄로 이유(왜 묻는지 / 답에 따라 무엇이 갈리는지)를 짧게 덧붙여, 사용자가 답의 무게를 판단할 수 있게 한다.
- 선택지가 있으면 각 옵션이 어떤 결과로 이어지는지 짧게 적는다. 자유 답변이면 인용 아래에 안내 한 줄.
- `What:` / `Why:` 같은 라벨 단어를 본문에 반복해 쓰지 않는다 — 헤딩과 인용의 시각적 구분이 그 역할을 대신한다.

### 3. 텍스트 우선, 도구는 보조

- 기본은 **평문 텍스트로 한 질문씩** 물어보는 방식. 사용자가 자연스럽게 답하기 좋고, 답이 다음 질문을 결정하는 흐름에 맞다.
- `AskUserQuestion` 도구는 선택지가 명확히 한정되어 있고 사용자가 빠르게 고르는 게 더 편한 경우에만 보조적으로 사용한다.
- 도구를 쓰더라도 "질문 + 이유 한 줄" 구조는 유지한다 (question 본문이 결정, description에 이유를 녹여넣음).

### 4. 사용자의 의도부터 정리

- 본격적인 질문에 들어가기 전, 내가 이해한 사용자의 요구를 **짧게 한 번 되짚는다** (1~3줄). 이미 합의된 부분을 다시 묻지 않기 위함.
- 되짚기 → 첫 질문(헤딩 + 인용 이유) → 답 수신 → 다음 질문 ... 순서.

### 5. 멈춰야 할 때

- 사용자가 "그냥 진행해", "알아서 해", "더 묻지 마" 같은 신호를 주면 이 모드를 즉시 중단하고 진행한다.
- 이미 답을 받은 항목을 다시 묻지 않는다. 같은 결정을 다른 표현으로 재질문하지 않는다.
- 답에 필요한 정보가 코드/파일에서 직접 확인 가능하면 사용자에게 묻기 전에 먼저 확인한다. 질문은 "사용자만이 답할 수 있는 것"에 한정한다.

### 이 모듈을 `apps/api/src/` 아래에 둘까요, 별도 패키지로 뺄까요?
> 다른 앱(예: admin)에서도 임포트할 가능성이 있으면 패키지, API 전용이면 모듈이 단순합니다.

- `apps/api/src/<name>` 모듈 — API 전용일 때
- `packages/<name>` 패키지 — admin 등 다른 앱에서도 쓸 때

### 나쁜 예 (한 번에 모두 던지기 / 이유 없음)

> 어디에 둘지, 이름은 뭘로 할지, DB 마이그레이션은 같이 할지, 테스트는 어떤 식으로 작성할지 알려주세요.

## 자동 트리거를 건너뛰는 경우

- 사용자가 이번 세션에서 이미 "묻지 말고 진행" 류의 지시를 했을 때.
- 질문할 필요 없이 코드/문맥에서 답을 찾을 수 있을 때 (먼저 확인하고, 그래도 모호하면 질문).
- 위험 없는 단순 작업으로 사용자의 의도가 이미 명확할 때 — 불필요한 질문은 오히려 흐름을 끊는다.
